# GameJam Unity 项目 Agent Harness

## 1. 项目概况

本项目是一个用于 GameJam 的 Unity 2D 游戏项目，目标是在极短开发周期内快速完成一个可玩、稳定、易迭代的游戏原型。

项目强调“先可玩，再好玩，最后再精致”。所有开发决策都应优先服务于核心玩法验证，不追求复杂架构、长期扩展性或过度抽象。

## 2. 技术栈约束

* Unity 版本：`Unity 2022.3.62f3c1`
* 渲染管线：URP
* 项目模板：2D 项目模板
* 主要语言：C#
* 默认平台：PC 本地运行优先
* UI 技术：优先使用 UGUI
* 输入系统：除非项目已有明确配置，否则优先使用 Unity 传统 Input Manager，避免临时引入复杂输入框架

## 3. Agent 工作原则

Agent 的首要目标是帮助项目快速形成可运行版本，而不是构建完美工程。

开发时必须遵守以下原则：

1. 优先完成最小可玩闭环。
2. 每次修改都应尽量保持项目可编译、可运行。
3. 不主动进行大规模重构。
4. 不为了“更优雅”而引入复杂设计模式。
5. 不实现当前需求之外的大型通用系统。
6. 不擅自引入第三方插件、包管理依赖或外部 SDK。
7. 不擅自修改项目设置、渲染管线配置、构建配置，除非任务明确要求。
8. 不修改 `.git`、`.gitignore`、版本控制配置相关内容，除非任务明确要求。
9. 遇到不明确需求时，优先做出最简单、最符合 GameJam 场景的合理实现。
10. 代码应服务于快速调试和快速迭代，允许适度硬编码，但必须集中、清晰、易改。

## 4. 开发优先级

开发时按以下顺序判断优先级：

1. 核心玩法能否跑通。
2. 玩家是否能理解并操作游戏。
3. 关键反馈是否清楚，例如命中、受伤、胜利、失败、拾取、交互。
4. 是否有明显 Bug、卡死、报错、无法继续游戏的问题。
5. 画面、音效、手感、节奏是否可以快速增强。
6. 代码结构是否足够清楚，方便继续迭代。

禁止为了第 6 点牺牲前 5 点。

## 5. 推荐目录结构

除非项目已有目录结构，否则优先使用以下结构：

```text
Assets/
  Art/
    Sprites/
    Animations/
    Materials/
  Audio/
    BGM/
    SFX/
  Prefabs/
    Characters/
    Enemies/
    UI/
    Effects/
    Environment/
  Scenes/
  Scripts/
    Runtime/
      Core/
      Player/
      Enemy/
      Gameplay/
      UI/
      Audio/
      Utils/
    Editor/
  ScriptableObjects/
  Settings/
```

目录使用应保持简单直观。不要为了分类而创建过深目录。

## 6. 代码规范

所有运行时代码放在 `Assets/Scripts/Runtime/` 下。

所有编辑器工具代码放在 `Assets/Scripts/Editor/` 下，并使用 `#if UNITY_EDITOR` 或放置于 Editor 目录，避免打包时报错。

C# 代码遵守以下规则：

1. 类名、方法名使用 PascalCase。
2. 私有字段使用 camelCase。
3. 可在 Inspector 配置的字段使用 `[SerializeField] private`。
4. 尽量避免 public 字段，必要时使用 public 属性或方法暴露。
5. 每个类顶部必须写 XML summary，说明该类的作用。
6. 核心 public 方法应写 summary，说明调用时机和用途。
7. 类内避免大段注释，只在关键逻辑处写短注释。
8. 单个脚本不要承担过多职责。
9. 不要把所有游戏逻辑塞进一个 `GameManager`。
10. 不要创建没有实际必要的接口、抽象基类、事件总线或框架层。

## 7. 组件设计原则

优先使用直观的 MonoBehaviour 组件完成需求。

推荐组件粒度：

* 玩家控制：`PlayerController`
* 玩家状态：`PlayerStatus`
* 敌人行为：`EnemyController`
* 伤害处理：`Health`
* 投射物：`Projectile`
* 拾取物：`PickupItem`
* 游戏流程：`GameFlowController`
* UI 显示：`UIController`
* 音频播放：`AudioController`

组件之间可以直接引用，GameJam 阶段不强制解耦。只要引用关系清楚、Inspector 可配置、调试方便即可。

## 8. 场景与 Prefab 约束

场景文件放在 `Assets/Scenes/` 下。

常用对象应制作成 Prefab，例如玩家、敌人、子弹、道具、UI 面板、特效。

修改 Prefab 时应保持 Inspector 字段完整，不要随意清空已有引用。

新增脚本后，如果需要挂载到对象上，必须说明应挂载在哪个对象、需要配置哪些字段。

## 9. UI 约束

UI 优先使用 UGUI。

UI 脚本应只负责显示、按钮事件和简单交互，不要把核心玩法逻辑写进 UI 脚本。

需要新增 UI 时，Agent 应说明需要提前准备的对象，例如：

* Canvas
* Button
* Text 或 TMP_Text
* Image
* Slider
* Panel
* 对应脚本挂载位置
* Inspector 字段绑定关系

如果项目未明确使用 TMP，可以优先使用 Unity UI Text；如果项目已有 TMP，则沿用 TMP。

## 10. 数据配置原则

少量配置可以直接写在 Inspector 字段中。

当同类数据数量较多时，可以使用 ScriptableObject，例如：

* 武器配置
* 敌人配置
* 道具配置
* 关卡波次配置
* 音效配置

不要为了少量数据提前设计复杂数据库、加载系统或 Addressables。

## 11. 音频约束

GameJam 阶段音频系统应保持简单。

推荐提供一个轻量 `AudioController`，支持：

* 播放 BGM
* 播放一次性音效
* 调整主音量
* 调整 BGM 音量
* 调整 SFX 音量

不要在没有明确需求时实现复杂的音频池、动态加载、混音快照系统或引用计数系统。

## 12. 动画与特效约束

角色动画优先使用 Animator。

简单 UI 动效、闪烁、缩放、淡入淡出可以直接使用 Coroutine 或简单插值实现。

不要在没有明确需求时引入复杂动画状态机、Timeline 或自定义动画框架。

特效优先使用 Sprite、Particle System、简单 Shader 或对象缩放透明度动画。

## 13. 物理与碰撞约束

2D 碰撞统一使用 Unity 2D 物理系统。

常用组件包括：

* Rigidbody2D
* Collider2D
* Trigger Collider2D

角色移动优先使用 Rigidbody2D 或 Transform 移动中的一种，不要混用导致物理表现异常。

涉及碰撞层时，应说明需要配置的 Layer 或 Tag。

## 14. 调试与日志约束

允许使用 `Debug.Log`、`Debug.LogWarning`、`Debug.LogError` 辅助调试。

日志必须有明确前缀，方便定位来源，例如：

```csharp
Debug.Log("[PlayerController] Jump triggered.");
```

不要在 `Update` 中持续刷屏输出日志，除非是临时调试，并且提交前应移除或关闭。

## 15. Agent 修改代码前的要求

Agent 在执行较大修改前，应先简要说明：

1. 要新增或修改哪些脚本。
2. 每个脚本负责什么。
3. 需要在 Unity Inspector 中配置什么。
4. 是否会影响已有功能。

如果只是小修 Bug，可以直接给出修改方案。

## 16. Agent 输出代码后的要求

Agent 输出代码后，必须补充说明：

1. 文件应放置的位置。
2. 脚本应挂载到哪个对象。
3. Inspector 中需要绑定哪些引用。
4. 如何在 Unity 中快速测试。
5. 是否存在临时写死的参数，以及之后如何调整。

## 17. 禁止事项

Agent 不得执行以下行为：

1. 不得修改 `.git` 目录。
2. 不得删除已有资源，除非任务明确要求。
3. 不得擅自重命名大量文件。
4. 不得擅自移动已有目录结构。
5. 不得引入未经确认的第三方库。
6. 不得将简单需求实现成复杂框架。
7. 不得在无需求时添加联网、账号、存档加密、热更新等系统。
8. 不得无说明地修改 Project Settings。
9. 不得输出无法直接编译的代码。
10. 不得假设场景中一定存在某个对象，除非明确说明需要创建。

## 18. 推荐实现风格

优先采用以下风格：

* 简单直接
* Inspector 可配置
* 易调试
* 易替换资源
* 少依赖
* 少继承
* 少全局状态
* 先完成，再优化

对于 GameJam 项目，能稳定跑起来的普通代码优先于理论上更优雅但难以调试的架构。

## 19. 最小可玩闭环检查表

在实现玩法时，应优先确认以下闭环是否完成：

* 玩家能开始游戏
* 玩家能移动或进行核心操作
* 游戏中存在明确目标
* 玩家操作能产生反馈
* 存在成功或失败条件
* 成功或失败后能重新开始或返回
* 游戏过程中没有明显报错
* 核心体验能在 1 分钟内被理解

## 20. 最终目标

本项目的最终目标不是完成一个工程完美的商业项目，而是在 GameJam 限制时间内完成一个可展示、可游玩、有核心趣味点的游戏原型。

所有代码、资源组织和功能实现，都应围绕这个目标服务。
